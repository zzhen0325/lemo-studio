# 参考图片优化与稳定保存方案 (本地优先 + 后台同步)

针对参考图片在历史记录中丢失的问题，我们将实施“本地持久化存储 + 后台静默同步”的方案。

## 1. 核心存储基础设施 (Persistence)
*   **本地数据库**: 创建 `lib/local-image-storage.ts`，利用 IndexedDB 持久化存储上传的图片文件（Blob）。
*   **优势**: 即使页面刷新，本地图片数据依然存在，不再依赖会话级的 Blob URL。

## 2. 数据模型升级 (Data Model)
*   **本地标识**: 在 `UploadedImage` 和 `Generation` 类型中增加 `localId` 字段。
*   **历史关联**: 历史记录在保存时将携带 `localId`。如果服务器路径尚未生成，UI 将通过 `localId` 从 IndexedDB 检索图片显示。

## 3. 优化上传与同步流程 (Sync Logic)
*   **即时响应**: 上传图片时，立即存入 IndexedDB 并显示本地预览。
*   **后台上传**: 异步启动 CDN 上传任务。
*   **自动替换**: 上传成功获取服务器 `path` 后，自动触发同步：
    1. 更新 Zustand Store 中的图片状态。
    2. **关键**: 发送请求给 `/api/history/sync-image`，将数据库中引用该 `localId` 的所有历史记录更新为永久的服务器路径。

## 4. 增强 UI 加载稳定性 (UI Stability)
*   **智能路径解析**: 修改图片显示逻辑，按优先级查找：服务器路径 (CDN) > 本地 IndexedDB 存储 > 内存 Blob URL。
*   **异常恢复**: 即使在上传中途刷新页面，应用重启后将继续完成未竟的上传和同步任务。

## 5. API 接口扩展
*   **同步接口**: 在 `app/api/history/route.ts` 中增加 `sync-image` 动作，支持批量更新历史记录中的源图路径。

***

**预期效果**: 用户上传图片后可立即开始生成，无需等待上传完成；历史记录中的参考图将永久稳定，不再受页面刷新的影响。
